# PSEO 改造方案 V2

## 核心思路

- 去除实时调用 LLM，改为数据预处理 + 静态内页
- 建立 tag 枚举体系，约束 LLM 输出，保证数据一致性
- 数据存 Cloudflare D1（SQLite 边缘数据库），通过数据管理 API（Workers）进行 CRUD
- 构建时从 D1 导出 JSON 供 Next.js 消费

---

## 〇、基础设施：数据管理服务

### 部署方案

数据管理小工具部署在 **Cloudflare Workers + D1**，与主站同一个 repo，独立目录。

```
japanesename/
├── src/                         ← 主站（Next.js + OpenNext）
├── tools/
│   └── data-service/            ← 数据管理服务（Cloudflare Workers + D1）
│       ├── src/
│       │   ├── index.ts         ← Hono 路由入口
│       │   ├── routes/          ← names, keywords, tags CRUD
│       │   └── db/
│       │       ├── schema.ts    ← Drizzle 表定义
│       │       └── migrate.ts   ← 迁移脚本
│       ├── wrangler.jsonc       ← D1 binding 配置
│       ├── package.json
│       └── drizzle.config.ts
├── shared/
│   └── types.ts                 ← 共享类型（NameRecord, KeywordRecord, TagEnum）
├── scripts/
│   ├── import-names.ts          ← 批量导入基础 name → 调 API 写入
│   ├── llm-enrich.ts            ← 取 20 条 → 调 LLM → 调 API 更新
│   └── build-export.ts          ← 调 API 导出 → 生成 Next.js 构建用 JSON
├── wrangler.jsonc               ← 主站的 wrangler 配置
└── package.json
```

### API 端点

```
GET/POST/PUT/DELETE  /api/names              ← name 库 CRUD
POST                 /api/names/query        ← filter_rule DSL 查询
GET/POST/PUT/DELETE  /api/keywords           ← keyword 库 CRUD
POST                 /api/keywords/coverage  ← 批量覆盖率检查（发布前验证匹配数）
GET                  /api/tags               ← tag 枚举查询
POST                 /api/export/:type       ← 导出 JSON（names / keywords / all）
```

### 为什么选 Cloudflare

| 维度 | Cloudflare Workers + D1 |
|------|------------------------|
| 部署 | `wrangler deploy` 一行搞定 |
| 存储 | D1 免费 5GB，20K 条记录约 20MB |
| 成本 | 已开 paid plan，零增量 |
| 可靠性 | 全球边缘 99.9%+，无需维护进程 |
| 与主站关系 | 同 Cloudflare 账号，独立 Workers 服务 |

### Code Agent 集成

不使用 MCP（Workers 不适合长连接），改为：
- 部署标准 REST API
- 编写 Cursor Skill 包装 API 调用
- Code agent 通过 Skill 直接操作数据（查名字、改标签、导出 JSON 等）

## 一、Tag 枚举库（核心基础设施）

> 所有 tag 维度和枚举值预先定义，LLM 填充 name 数据时只能从中选取。
> 存储方式：D1 `tag_dimensions` 表 + 导出为 `tag-registry.json` 供 LLM prompt 和前端使用。

### 维度定义

| 维度 | 字段名 | 类型 | 枚举值 | 说明 |
|------|--------|------|--------|------|
| 性别 | `gender` | 单选 | `male`, `female`, `unisex` | 姓名适用性别 |
| 名字类型 | `name_part` | 单选 | `given_name`, `family_name` | 名 or 姓 |
| 年代 | `era` | 单选 | `ancient`, `traditional`, `modern`, `2000s`, `trending` | 名字流行的时代 |
| 流行度 | `popularity` | 单选 | `very_common`, `common`, `uncommon`, `rare`, `unique` | 在日本的使用频率（可由 estimated_population 自动推导） |
| 书写体系 | `script` | 多选 | `kanji`, `hiragana`, `katakana`, `romaji` | 该名字使用的书写方式 |
| 来源 | `origin_region` | 单选 | `japan_native`, `japanese_american`, `international` | 名字的文化来源 |
| 地域 | `regional_origin` | 文本 | 自由文本（okinawa, kanto, kansai…） | 名字主要分布/起源地域（主要 family_name） |
| 使用场景 | `use_case` | 多选 | `real_person`, `anime`, `samurai`, `warrior`, `pet`, `game_character`, `baby`, `fantasy`, `historical`, `vtuber` | 名字常见的使用场景 |
| 气质 | `vibe` | 多选(≤3) | `cute`, `cool`, `elegant`, `strong`, `mysterious`, `dark`, `gentle`, `noble`, `playful`, `fierce`, `serene`, `warm`, `edgy`, `scary`, `funny` | 名字给人的感觉 |
| 主题元素 | `element` | 多选(≤3) | `fire`, `water`, `ice`, `light`, `dark`, `wind`, `earth`, `moon`, `sun`, `flower`, `mountain`, `sky`, `star`, `death`, `blood`, `thunder` | 名字蕴含的意象 |
| 英语音节 | `syllable_count` | 数值范围 | `1`-`6` | 罗马音音节数（1-2=short, 3=medium, 4+=long） |
| 日语モーラ | `mora_count` | 数值范围 | `1`-`8` | 日本語モーラ数，从假名 reading 计算 |
| 汉字数 | `kanji_count` | 数值范围 | `0`-`4` | 名字中的汉字数（0=纯假名，1=单字名，2-3=常见） |
| 世帯数 | `household_count` | 数值 | — | 世帯数（来自 myoji-yurai.net 等，仅 family_name） |
| 推定人口 | `estimated_population` | 数值 | — | 推定人口数（如 新垣≈34000） |
| 全国排名 | `national_rank` | 数值 | — | 全国姓氏排名（如 新垣=563，仅 family_name） |
| 汉字含义 | `kanji_meaning_tags` | 多值开放（≥10） | 不限枚举，LLM 尽量多打。覆盖汉字本义 + 引申义 + 联想义 + 颜色 + 情感 | 万能兜底字段，覆盖 vibe/element 不到的语义 |
| 逐字释义 | `kanji_breakdown` | JSON 数组 | `[{"kanji","meanings_en","reading"}]` | 每个汉字的含义拆解，内页核心内容区 |
| 异读 | `alternative_readings` | JSON 数组 | 假名列表 | 同一汉字的其他读法（来自 myoji-yurai.net） |
| 罗马字变体 | `reading_romaji_variants` | JSON 数组 | 罗马字列表 | 长音转写差异等（Ōno / Ohno / Oono） |
| 起源由来 | `etymology_en` | 文本 | — | 名字的历史/地理起源说明 |
| 有名人 | `famous_bearers` | JSON 数组 | `[{"name","context"}]` | 知名人物列表 |
| 家紋 | `kamon_url` | 文本 | URL | 家紋图片 URL（仅 family_name） |

### Tag 示例

| 维度 | 值 | 示例名字 |
|------|------|---------|
| gender=male | 太郎/Tarō | gender=female | 花子/Hanako | gender=unisex | 光/Hikaru |
| era=ancient | 信長/Nobunaga | era=modern | 蓮/Ren | era=2000s | 大翔/Hiroto |
| vibe=cute | さくら/Sakura | vibe=dark | 闇/Yami | vibe=strong | 剛/Tsuyoshi |
| use_case=anime | 鬼丸/Onimaru | use_case=pet | モチ/Mochi | use_case=samurai | 武蔵/Musashi |
| element=fire | 炎/Honō | element=light | 光/Hikaru |
| popularity=very_common | 佐藤/Satō | popularity=rare | 毒島/Busujima |

### Tag 库管理原则

1. **枚举值只增不改**：新增值直接追加；已有值不能重命名或删除（除非迁移）
2. **LLM prompt 约束**：调用 LLM 填充 name 标签时，prompt 中必须附带完整枚举列表，要求 LLM 只能从中选取
3. **校验层**：LLM 返回结果入库前，程序校验所有 tag 值是否在枚举范围内，不在的标记为异常人工审核

## 二、Name 库

### 存储方式
- Cloudflare D1（SQLite 边缘数据库）
- 预计 10000+ 条
- 多选字段（use_case, vibe, element 等）以 JSON 数组存储，D1 支持 `json_each()` 查询

### Schema

```json
{
  "id": "gn_yui_7d93",
  "romaji": "Yui",
  "kanji": "結衣",
  "reading": "ゆい",
  "reading_romaji_variants": ["Yui"],

  "gender": "female",
  "name_part": "given_name",

  "era": "modern",
  "popularity": "very_common",
  "syllable_count": 2,
  "script": ["kanji", "hiragana"],
  "origin_region": "japan_native",

  "use_case": ["real_person", "anime", "baby"],
  "vibe": ["cute", "gentle", "elegant"],
  "element": ["love"],
  "kanji_meaning_tags": ["tie", "bind", "clothing", "garment"],

  "meaning_en": "To tie or bind together with clothing; symbolizes connection and protection",
  "meaning_zh": "将衣物系在一起；象征连结与守护",
  "description_en": "A beloved modern Japanese girl's name popularized by actress Yui Aragaki. The kanji 結 (tie/bind) paired with 衣 (clothing) evokes the image of wrapping someone in warmth and connection.",
  "description_zh": "一个深受喜爱的现代日本女性名字，因演员新垣结衣而广为人知。結（系、绑）搭配衣（衣服），唤起温暖与连结的意象。",
  "famous_bearers": [
    { "name": "Aragaki Yui", "context": "actress" },
    { "name": "Horie Yui", "context": "voice actress / singer" }
  ],
  "related_names": ["Yuki", "Mei", "Yuna"],

  "status": "complete",
  "created_at": "2026-03-20T00:00:00Z",
  "updated_at": "2026-03-20T00:00:00Z"
}
```

### 相比原版的改动说明

| 字段 | 变动 | 理由 |
|------|------|------|
| `meaning_zh` / `description_zh` | 新增 | 支持中文 locale |
| `famous_bearers` | 改为数组 | 一个名字可能有多个知名人物 |
| `related_names` | 新增 | 内页交叉推荐用 |
| `reading_romaji_variants` | 新增 | 部分名字有多种罗马音拼法（如 Ōno / Ohno / Oono） |
| `status` | 新增 | 追踪数据处理进度：`raw` → `llm_enriched` → `reviewed` → `complete` |
| `created_at` / `updated_at` | 新增 | 数据管理基础字段 |

### Name 库处理流程

```
1. 导入基础 name 数据（romaji, kanji, reading, gender, name_part）
   来源：公开姓名数据集、爬取
   状态 → raw

2. 批量 LLM 填充（每批 20 条）
   输入：基础字段 + tag-registry.json（完整枚举表）
   输出：era, popularity, use_case, vibe, element, meaning, description 等
   状态 → llm_enriched

3. 程序校验
   检查所有 tag 值是否在枚举范围内
   异常数据标记 → 人工审核

4. 人工抽检 / 修正
   状态 → reviewed → complete
```

## 三、Keyword 库

### 存储方式
- Cloudflare D1（与 name 库同一数据库）
- Google Sheet 可作为录入/协作入口，通过脚本调 API 同步入库
- 构建时通过 `/api/export/keywords` 导出 JSON 供 Next.js 使用

### Schema

```json
{
  "id": "kw_female_001",
  "keyword": "female japanese names",
  "search_volume": 12100,
  "kd": 51,
  "intent": "informational",

  "strategy": "category_page",
  "strategy_note": "做一个女性名字分类内页，展示女性名字列表和选择引导",
  "path": "/names/female",
  "priority": 1,
  "status": "draft",

  "seo": {
    "title": "200+ Beautiful Female Japanese Names with Meanings",
    "description": "Discover elegant female Japanese names for your baby, character, or story...",
    "h1": "Female Japanese Names",
    "h2": ["Popular Female Names", "Traditional Female Names", "Cute Female Names", "Names by Meaning"],
    "keywords": ["female japanese names", "japanese girl names", "japanese names for girls"]
  },

  "filter_rule": {
    "must": {
      "gender": "female",
      "name_part": "given_name"
    },
    "should": {}
  },

  "quiz": [
    {
      "question_en": "What style are you looking for?",
      "question_zh": "你想要什么风格？",
      "dimension": "vibe",
      "options": ["cute", "elegant", "cool", "gentle"]
    },
    {
      "question_en": "What era do you prefer?",
      "question_zh": "你偏好什么年代？",
      "dimension": "era",
      "options": ["modern", "classical", "contemporary"]
    },
    {
      "question_en": "What theme resonates with you?",
      "question_zh": "什么主题与你共鸣？",
      "dimension": "element",
      "options": ["nature", "love", "beauty", "strength", "light"]
    }
  ],

  "related_keywords": ["kw_cute_female_001", "kw_girl_001", "kw_baby_girl_001"],

  "created_at": "2026-03-20T00:00:00Z",
  "updated_at": "2026-03-20T00:00:00Z"
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `strategy` | 该关键词的用途分类：`homepage_seo`（优化首页）、`category_page`（分类内页）、`blog_post`（博客文章）、`tool_page`（工具页，如名字转写器） |
| `path` | 对应的页面路径 |
| `priority` | 优先级（1-5），用于控制构建批次。建议先做 search_volume > 5000 的关键词 |
| `status` | `draft` → `seo_ready`（SEO 字段已填） → `filter_ready`（匹配规则已设） → `quiz_ready`（问答已配） → `published` |
| `filter_rule` | Tag 匹配规则，采用 `field/op/value` DSL，`must` 为必须满足，`should` 为加分排序 |
| `quiz` | 该内页的选择题配置，每题对应一个 tag 维度，选项从 tag 枚举库中取 |
| `related_keywords` | 关联页面，每个带 label、path 和独立的 filter_rules（用于内链推荐） |
| `seo_guidance` | 自然语言 SEO 指导（描述该页面应怎么做） |

### Filter Rule DSL

统一采用 `field` / `op` / `value` 格式。同时支持 **tag 过滤**和**字段条件查询**，覆盖所有类型的关键词。

#### 运算符

| 运算符 | 含义 | 适用类型 | SQL 映射 |
|--------|------|---------|---------|
| `eq` | 严格等于 | enum 单选 | `field = value` |
| `any_of` | 命中任一（OR） | enum / multi_enum (JSON) | `field IN (...)` / `json_each` |
| `all_of` | 必须全部包含 | multi_enum (JSON) | `json_each` + COUNT |
| `gte` | >= | 数值 | `field >= value` |
| `lte` | <= | 数值 | `field <= value` |
| `gt` | > | 数值 | `field > value` |
| `lt` | < | 数值 | `field < value` |
| `starts_with` | 前缀匹配 | 字符串（romaji） | `field LIKE 'value%'` |
| `contains` | 包含子串/元素 | 字符串 / JSON 数组 | `field LIKE '%value%'` / `json_each` |

#### 示例

```jsonc
// 示例 1：anime 名字
{
  "must": [
    { "field": "use_case", "op": "any_of", "value": ["anime", "game_character"] }
  ],
  "should": [
    { "field": "vibe", "op": "any_of", "value": ["cool", "mysterious", "fierce", "edgy"] },
    { "field": "era", "op": "any_of", "value": ["modern", "2000s"] }
  ]
}

// 示例 2：cute female names
{
  "must": [
    { "field": "gender", "op": "any_of", "value": ["female", "unisex"] },
    { "field": "vibe", "op": "any_of", "value": ["cute", "playful"] }
  ],
  "should": [
    { "field": "name_part", "op": "eq", "value": "given_name" }
  ]
}

// 示例 3：names starting with K（非 tag 查询）
{
  "must": [
    { "field": "romaji", "op": "starts_with", "value": "K" }
  ],
  "should": [
    { "field": "popularity", "op": "any_of", "value": ["common", "very_common"] }
  ]
}

// 示例 4：short japanese names（音节数查询）
{
  "must": [
    { "field": "syllable_count", "op": "lte", "value": 2 }
  ],
  "should": []
}

// 示例 5：names meaning death（kanji_meaning_tags 兜底）
{
  "must": [
    { "field": "kanji_meaning_tags", "op": "any_of", "value": ["death", "perish", "underworld", "dark", "shadow"] }
  ],
  "should": [
    { "field": "vibe", "op": "any_of", "value": ["dark", "mysterious"] }
  ]
}
```

#### 查询逻辑

```
must  → 决定名字是否出现在页面上（AND 关系）
should → 决定排序优先级（命中越多排越前，同分随机）
```

### related_keywords 格式

每个关联页面自带独立的 filter_rules，不依赖关键词 ID：

```json
{
  "label": "Female Japanese Names",
  "path": "/names/female",
  "filter_rules": [
    { "field": "gender", "op": "any_of", "value": ["female", "unisex"] },
    { "field": "name_part", "op": "eq", "value": "given_name" }
  ]
}
```

## 三-B、Tag 覆盖率保障

> 核心问题：某些关键词的 filter_rule 可能只匹配到极少量名字，导致页面内容单薄。
> 以下策略从数据源、查询层、渲染层三个环节防范。

### 1. `kanji_meaning_tags` 作为万能兜底

`kanji_meaning_tags` 是**唯一不受枚举约束**的 tag 字段。LLM 打标签时规则：

- **不限数量**，尽可能多打（10-20 个都可以）
- 覆盖：汉字本义 + 引申义 + 联想义 + 颜色 + 情感
- 例如「紅」→ `["red", "crimson", "passion", "fire", "warmth", "celebration"]`
- 例如「鬼」→ `["demon", "ghost", "oni", "scary", "fierce", "supernatural"]`

当 vibe/element 等枚举覆盖不到某个关键词语义时（如 "names meaning red"），用 `kanji_meaning_tags` 兜底查询。

### 2. 发布前覆盖率检查

keyword 从 `filter_ready` → `published` 前，必须跑匹配验证：

```
执行 filter_rule 查询 name 库 → 统计匹配数 N

N >= 50     → ✅ 正常发布 (category_page)
N = 20~49   → ⚠️ 可发布，标记 thin_risk，考虑放宽 filter 或补数据
N = 5~19    → 🔄 降级为 blog_post（长文内容 + 少量名字示例）
N < 5       → ❌ 暂不发布，加入 needs_data 队列
```

在数据管理 API 中提供批量检查端点：
```
POST /api/keywords/coverage-check
→ 返回每个 keyword 的匹配数量和建议策略
```

### 3. 渐进放宽展示

页面渲染时，如果精确匹配数量不够，自动向外扩展：

```
┌─ 第一层：must 全部命中         → "Best Matches"（精选）
├─ 第二层：去掉最窄的 must 条件  → "Similar Names"（相似）
└─ 第三层：只保留 gender/name_part → "More to Explore"（更多）
```

确保每个页面至少有 50+ 条名字展示，SEO 不会因为内容单薄被降权。

### 4. vibe 枚举按需补充

观察 keyword 需求驱动，当前缺失但有搜索量的 vibe 值：

| 候选值 | 对应关键词 | 搜索量 | 是否加入 |
|--------|-----------|--------|---------|
| `scary` | "scary japanese names" | ~590 | 建议加 |
| `funny` | "funny japanese names" | ~590 | 建议加 |
| `romantic` | 暂无直接关键词 | - | 暂不加 |

原则：**有关键词需求 + 能稳定匹配 20+ 名字 → 加入枚举**。否则用 `kanji_meaning_tags` 兜底。

## 四、内页生成 SOP

以 `female japanese names` (path: `/names/female`) 为例：

### 构建阶段（build time）

```
Step 1  读取 keyword 记录
        → 拿到 seo 字段、filter_rule、quiz 配置、related_keywords

Step 2  调用数据管理 API 执行 filter_rule 查询
        → POST /api/names/query { filter_rule, limit: 200 }
        → D1 内部: SELECT * FROM names WHERE gender='female' AND name_part='given_name'
        → 按 should 规则排序，取 Top 200 条

Step 3  组装页面数据 JSON
        {
          seo: { title, description, h1, h2, keywords },
          names: [ ... 200条 name 数据 ... ],
          quiz: [ ... 3-5 道选择题 ... ],
          related: [ { keyword, path, title } ... ]
        }

Step 4  导出到 /data/keywords/female.json
```

### 渲染阶段（Next.js）

```
/names/[slug]/page.tsx

- generateStaticParams() → 读取所有 keyword 的 path，生成静态路由
- generateMetadata()     → 从 keyword 数据填充 title, description, keywords, canonical
- Page 组件：
  ├── SEO Header（H1, H2, 介绍文案）        ← 静态，爬虫可见
  ├── Name List（名字卡片列表，分页/分组）    ← 静态，爬虫可见
  ├── Quiz Section（选择题交互）              ← 客户端 JS，交互增强
  │   └── 用户选择后，前端从已加载的 200 条中过滤
  ├── Related Keywords（内链推荐区）          ← 静态，传递权重
  └── FAQ Schema（JSON-LD structured data）  ← SEO 加分
```

### 交互阶段（client-side）

**随机推荐**：页面加载时展示前 N 条 + "换一批" 按钮（前端从 200 条中随机）

**选择题引导**：
1. 用户依次回答 3 道题（vibe / era / element）
2. 前端用选中的 tag 值，在已加载的 200 条 name 中做客户端过滤
3. 展示匹配结果（通常 5-20 条）
4. 无需任何 API 调用，纯前端过滤

## 六、Sitemap & 内链策略

### Sitemap
- 构建时自动生成，包含所有已 published 的 keyword 页面
- 按语言生成 `hreflang` 标记
- 大于 50000 条时拆分为 sitemap index

### 内链
- 每个内页底部展示 4-8 个 related keywords 的链接卡片
- 首页展示热门分类入口（search_volume Top 10 的关键词）
- Name 卡片可以链接到该 name 出现的其他分类页（交叉引用）
